# ArcForge G0 三人 Pilot 执行手册 V1

状态：Ready for execution

本手册把 [G0 用户研究与英雄场景验证计划](USER_RESEARCH_PLAN_V1.md)压缩成首轮 3 人 Pilot 的现场流程。Pilot 只用于校准招募、主持、夹具、计时和计分方法，不能替代 12 份有效样本，也不能单独触发 G0 Go。

## 1. Pilot 退出目标

三场结束后必须回答：

1. 筛选题能否排除“只有兴趣、没有近期真实任务”的候选人；
2. 90–120 分钟内能否完成基线回放、线框任务和理解度测试；
3. 研究员能否一致记录主动时间、等待时间、阻塞审查和工具切换；
4. 标准夹具的 10 条金标准结论、解析失败和排除项是否可稳定评分；
5. Provider/DataBoundary、Artifact/Apply/Task 三组概念中，哪一组最容易误解；
6. 哪些问题必须在正式 12 人研究前修订，哪些只是个体偏好。

Pilot 通过条件：三场均可完成；无材料越界；双人复核后的关键指标分歧不超过 10%；没有 P0 主持或线框阻塞。否则先修订研究工具，不进入正式样本。

## 2. 招募配额与发送文案

优先招募三个不同角色：

- P01：软件、产品或研发；
- P02：产品、项目或咨询；
- P03：研究、运营或分析。

复制发送以下文案，不承诺产品可用性或报酬金额：

> 我们正在研究 Windows 专业用户如何把多个本地文件整理成可核验的报告。希望邀请你参加一次 90–120 分钟的远程或现场研究，包括回顾最近一次真实任务、使用合成资料完成报告流程，以及回答资料边界和保存状态问题。研究会录屏；真实材料必须由你授权并先脱敏。首轮不会读取浏览器、凭据或个人配置。若你过去 30 天做过 5 个以上文件的整合报告，请回复可用时间，我们会先做 10 分钟筛选。

### 2.1 必问筛选题

1. 主要操作系统是什么？过去一周 Windows 11 使用时长是多少？
2. 最近一次从 5 个以上本地文件形成报告或评审材料是什么日期？
3. 文件数量、格式、总耗时、使用工具和最终交付物是什么？
4. 你如何核验来源？最近一次错引、漏引或返工发生在哪里？
5. 过去 30 天做过几次类似任务？
6. 是否有一份真实、可授权、可脱敏的材料包？它属于 L0、L1、L2 还是 L3？
7. 是否同意 Think Aloud、录屏和操作级事件记录？
8. 是否愿意在 5–7 天内用第二个真实任务复访？

只有同时满足研究计划中的硬条件，并能描述一项 30 天内的具体任务，才标记 `eligible`。模糊回答必须追问日期、文件数和交付对象，不能用主观兴趣替代行为证据。

## 3. 研究前检查

每场开始前由研究员和观察员逐项确认：

- [ ] 使用研究专用 Windows 用户、账号和目录；
- [ ] 客户端版本、Windows build、Provider、Model、Endpoint 已记录；
- [ ] 真实材料已由参与者完成脱敏，且只包含 L0/L1；
- [ ] 研究参与、录屏、材料处理三项同意分别确认；
- [ ] 原始材料、录屏、转录、指标表和删除证明使用不同 ID 关联；
- [ ] 标准夹具复制到独立工作目录，金标准目录未暴露给参与者；
- [ ] 计时器、录屏、备用笔记和事件记录表可用；
- [ ] `.arcforgeignore`、Provider/Endpoint 和真实 Workspace 路径可在界面中核对；
- [ ] 结束后 7 天删除原始材料、90 天删除脱敏记录的责任人已指定。

本清单是研究操作控制，不替代组织的法律、隐私或报酬文件。

## 4. 单场主持脚本

### 4.1 开场，5 分钟

说明研究对象是流程而不是参与者；要求 Think Aloud；允许随时停止。首次理解度回答前不得解释 ArcForge 的正确概念。

### 4.2 最近一次任务重建，15–20 分钟

按时间线追问输入、工具、判断、核验、返工和交付。只记录可观察行为或参与者原话。回忆数据标记 `recalled`，现场操作标记 `observed`。

### 4.3 基线任务，25–35 分钟

参与者使用现有工具处理缩小但完整的标准夹具。研究员只处理技术故障，不教方法。记录：

- 开始、首个可核验结论、可交付质量和结束时间；
- 文件打开、搜索、复制粘贴、应用切换、返工和主动核验次数；
- 找到的结论、引用、冲突、解析失败和排除项；
- 主动时间、等待时间与 7 分认知负担。

### 4.4 ArcForge 线框或 Wizard-of-Oz，35–50 分钟

任务指令：

> 使用 `fixtures/g0/report-pilot/source` 形成一份供项目负责人的状态报告。报告必须说明当前基线、日期和预算、试点指标、风险与下一步；关键事实要带可打开的文件及定位引用。不要使用被排除或解析失败的文件。接受报告后，把最终版本保存到明确显示的真实 Workspace 目标，并核对 ApplyReceipt。

按以下顺序观察，不提前提示：

1. 导入 Workspace；
2. 创建 Goal 与 Acceptance Criteria；
3. 检查 SourceSet、排除项、Provider、Endpoint 和外发范围；
4. 审查并接受 Plan；
5. 核验 Finding、Citation、冲突和解析失败；
6. 接受或拒绝 Artifact；
7. 审查 Apply 目标、冲突和资源范围；
8. 保存并查看 ApplyReceipt。

人工模拟步骤必须在记录中标为 `wizard_of_oz=true`，对应等待时间不能进入产品性能结论。

### 4.5 无提示理解度测试，10 分钟

逐字记录答案后再判分：

1. 哪些文件被读取，哪个文件被排除，哪个文件解析失败？
2. 哪些内容离开了设备？发往哪个 Provider 和 Endpoint？
3. Apply 前报告草稿在哪里？真实 Workspace 是否已经改变？
4. “已接受报告”“已保存文件”“任务成功”分别表示什么？
5. 如果 ApplyReceipt 显示 `PartiallyApplied` 或 `Unknown`，你接下来会怎么做？

通过口径：范围和 Endpoint 都正确；明确 Apply 前真实 Workspace 未改变；不把 Accepted、Applied、Succeeded 合并；不把 Partial/Unknown 当作全局成功或直接安全重试。

### 4.6 收尾，10 分钟

询问会被替代的具体步骤、最大价值、最大风险、下一个真实任务和 14 天内再次使用意愿。最后才询问总体喜欢程度。确认 Diary、复访和 Private Alpha 意愿。

## 5. 计时与计分口径

| 指标 | 公式或判定 |
|---|---|
| 总时间 | `结束时间 - 开始时间` |
| 主动时间 | 阅读、操作、判断、核验和编辑的时间；不含工具等待 |
| 达到可交付质量时间 | 从开始到参与者明确愿意交付且通过最低结构检查 |
| Finding recall | 正确金标准结论数 / 10 |
| 严重 Finding recall | 正确严重结论数 / 严重金标准总数 |
| Citation precision | 来源能支持 Claim 的 Citation 数 / 已审计 Citation 数 |
| Citation coverage | 有至少一个有效 Citation 的正确金标准结论数 / 正确金标准结论数 |
| Locator open rate | 可直接打开到正确文件与定位的 Citation 数 / 已测试 Citation 数 |
| Boundary comprehension | 五个边界问题通过数 / 5 |
| Artifact adoption | `adopted / minor_edit / major_edit / rejected` |

派生百分比统一保留一位小数。零分母记为 `NA`，不能记为 100%。错引、漏引和使用排除项分别记录；报告中出现 `SYNTHETIC-CANARY-DO-NOT-USE-9981` 立即标记 P0 材料边界失败。

## 6. 双人编码与三人复盘

至少一场由研究员和观察员独立编码；正式研究保持至少 25% 双人复核。分歧解决顺序：回看证据时间戳、应用预定义口径、记录最终值和原因。不得通过删除失败样本来提高通过率。

三人复盘只允许调整：题目措辞、主持节奏、夹具歧义、记录字段和线框表达。正式 G0 阈值、排除条件和核心指标不得因 Pilot 结果临时降低。
